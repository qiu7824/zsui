# ZSUI v0.2: Native UI Application Closure

## Goal

ZSUI v0.2 is not a window-creation milestone. Its goal is to make a useful,
independent Rust-first UI framework that can build complete lightweight native
desktop applications from one shared application codebase.

The same Rust `State`, `Msg`, `view`, `update`, theme, layout, widget tree and
business services must run through real Win32, AppKit and GTK4 backends. Platform
code is limited to host, renderer, input adapter and native-service modules.
Android and Harmony runtime work is outside this milestone.

The one-line entry point is only the application bootstrap contract:

```rust
fn main() -> zsui::ZsuiResult<()> {
    zsui::native_window("Application")
        .size(960, 640)
        .stateful_view(AppState::default(), view, update)
        .run()
}
```

Opening a native window alone does not complete a platform or this milestone.

## Framework Closure

ZSUI v0.2 must close all of these layers as one system.

### Application runtime

- RAII application and window lifecycle
- typed `State`, `Msg`, `Command`, `WidgetId` and `WindowId`
- deterministic `view` and `update` message flow
- explicit effects, timers, asynchronous task completion and invalidation
- `Result<T, ZsuiError>` failures without public backend pointers

### Layout, widgets and composition

- typed `Dp`, `Px`, `Dpi`, constraints, measurement and arrangement
- row, column, stack, grid, overlay, clipping and responsive constraints
- complete widget states: normal, hover, pressed, focused, selected and disabled
- text, button, icon button, text box, text area, check box, radio button,
  toggle, slider, progress, tabs, menu, dialog and tooltip
- list, scroll view and virtual/paged list with stable focus and scroll anchors
- reusable application compositions: navigation shell, command bar, grouped
  settings cards, setting rows, document shell and action areas

A declaration, catalog entry or empty paint path is not an implemented widget.
An implemented widget has layout, input, state, paint, theme and tests.

### Rendering and visual system

- retained widget tree converted to backend-neutral draw plans
- antialiased text and geometry, clipping, opacity and buffered repaint
- damage/invalidation rules that do not flash during resize or scrolling
- semantic theme tokens for typography, color, spacing, radius and elevation
- light, dark, high-contrast and system-theme reactions
- semantic icons resolved through Segoe Fluent/MDL2, SF Symbols and GTK icon
  themes, with an optional licensed fallback asset set

### Input and accessibility

- pointer, wheel, touchpad and keyboard event conversion
- hit testing, capture, focus order, Tab traversal and keyboard activation
- selection, caret, clipboard editing and undo/redo foundations
- native text input and basic CJK IME composition without crashes or lost text
- DPI/Retina/scale-factor relayout and accessible names, roles and states

### Native desktop services

- menus and context menus
- text clipboard
- open/save file dialogs
- system theme and scale notifications
- tray, global shortcuts, notifications and drag/drop where capabilities permit
- explicit capability results for unsupported or partial platform behavior

### Rust build and AI development experience

- safe public API; backend `unsafe` remains private and narrowly documented
- composition and traits instead of control inheritance or reflection
- no global mutable widget registry
- widgets and heavy native dependencies remain optional Cargo features
- default features stay small; feature combinations have automated checks
- `docs/ai-agent.md` and context packs let an agent load only the relevant
  architecture and files instead of scanning the whole repository
- examples demonstrate framework APIs but do not dictate framework boundaries

## Stable Platform Boundary

The safe Rust desktop boundary includes:

- `DesktopHost`
- `WindowService`
- `ClipboardService`
- `FileDialogService`
- `MenuService`
- `ThemeService`
- `TextInputService`

Every operation has an explicit `DesktopCapability`. `Partial` and
`Unsupported` are not runtime success. Raw `HWND`, Objective-C objects, GTK
pointers, drawing handles and platform event-loop types remain private.

Backend selection uses independent optional features:

- `windows-win32`
- `macos-appkit`
- `linux-gtk`
- `desktop-native` for target-selected desktop dependencies

## Real Backends

### Windows

Keep the proven Win32 host and buffered drawing path. Complete DPI, IME,
accessibility, menus, clipboard, dialogs, system theme, focus, drag/drop and
resize repaint without regressing background erase suppression, antialiasing,
spacing or scrolling.

### macOS

Use a real `NSApplication`/`NSWindow` host, AppKit event loop, Retina scaling,
`NSMenu`, `NSPasteboard`, `NSOpenPanel`, `NSSavePanel`, text input and SF
Symbols. Winit transport is not AppKit completion evidence.

### Linux

Use a real `GtkApplication`/`GtkApplicationWindow` host, GTK event controllers,
GIO/GTK menus, file chooser, clipboard, input method and icon theme. Validate
Wayland and retain basic X11 compatibility. GTK types do not enter widget APIs.

## Acceptance Applications

`examples/desktop_native_showcase.rs` is the cross-platform interaction probe.
It uses one application entry point and exercises navigation, command bar,
single-line and multiline text, buttons, toggle, list scrolling, theme,
open/save, copy/paste, native menu, resize, scale and keyboard focus.

The notepad, calculator, settings, workbench and paged-list examples are
secondary acceptance probes. Their application code should become smaller as
framework compositions mature, but demo line count is not the architecture
target and support-project dependencies are not committed.

## Evidence Gate

A capability or backend advances only with all applicable evidence:

1. Contract: safe API, capability semantics and no backend-type leakage.
2. Unit: layout, state machine, message routing, focus, theme and error tests.
3. Compile: formatting, no-default tests, full tests, feature matrix and target
   backend/example checks on Windows, macOS and Ubuntu stable Rust.
4. Runtime: startup, light/dark, text input, menu, dialog and resize captures
   under `docs/platform-proof/<platform>/`.
5. Interaction: lifecycle, typed messages, selection/copy/paste, CJK IME,
   scrolling, Tab focus, open/save, theme change and artifact-free resize.

`scripts/check-desktop-proof.ps1 -RequireComplete` is the final platform gate.
Placeholders, declarations, mocks and cross-compilation alone cannot satisfy it.

## Current Baseline

| Area | Status | Meaning |
| --- | --- | --- |
| Rust-first core/runtime | In progress | Typed messages, view runtime, errors, units and feature manifests exist; effects and accessibility remain incomplete |
| Layout/widgets | In progress | Common controls and reusable shells have first-pass runtime surfaces; many WinUI families remain contract-only or unstarted |
| Rendering/input | Three native renderers with first-pass typed input, focus visuals, IME and resize relayout | Win32 buffered paint plus AppKit NSView and GTK4 Cairo/Pango consume the shared draw plan; all three route click, scroll, focus, a semantic accent focus ring, keyboard activation, committed text and provisional IME preedit, while actual AppKit/GTK content bounds rebuild layout and hit geometry; precise caret/selection behavior and target proof remain incomplete |
| Native services | In progress | Win32 HMENU/HACCEL routing is connected; AppKit and GTK4 owned window, UTF-8 clipboard, open/save and typed native menu adapters are target-native slices |
| Windows Win32 | First pass | Real host and drawing path exist; complete proof and remaining services are pending |
| macOS AppKit | Native host/render/input first pass | Unified run/smoke enter NSApplication with an owned NSView/NSTextInputClient routing resize relayout, mouse, scroll, focus, keyboard, marked text and UTF-8 commits; precise selection, resize screenshot and target interaction proof remain incomplete |
| Linux GTK4 | Native host/render/input first pass | Unified run/smoke enter GtkApplication with a Cairo/Pango DrawingArea/GtkIMMulticontext routing allocation relayout, click, scroll, focus, keyboard, preedit and UTF-8 commits; precise selection, resize screenshot and Wayland/X11 interaction proof remain incomplete |
| AI/context tooling | First pass | Context packs and validation exist; they must track every new ownership boundary |
| Three-platform proof | Not complete | Complete target artifacts and interaction reports do not yet exist |

## Execution Order

Work proceeds in complete vertical slices, not by declaring many empty widgets:

1. Keep core runtime, draw plan, event and capability contracts stable.
2. Finish one high-frequency application path on Win32 without visual regressions.
3. Carry the same path through real AppKit and GTK4 adapters.
4. Add unit, feature-matrix, target compile and runtime evidence.
5. Update the component catalog and completion percentage only after evidence.
6. Continue with the next highest-value path: text/IME, focus, scroll/list,
   menu/command, theme/icon, dialogs, accessibility, then additional widgets.

ZSUI v0.2 completes only when one shared nontrivial application is genuinely
usable on all three native backends and the framework layers above are verified
as a coherent whole. A native window, a large component catalog or a polished
Windows-only demo cannot complete the milestone by itself.
