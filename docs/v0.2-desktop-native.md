# ZSUI v0.2: Native Desktop Closure

## Objective

ZSUI v0.2 will run one Rust `State`, `Msg`, `view`, `update`, theme, layout,
control tree and file-I/O layer on Windows, macOS and Linux. Platform code is
limited to host, renderer and native-service modules.

Android and Harmony runtime work is outside this milestone.

The acceptance application is
`examples/desktop_native_showcase.rs`. It must use one application entry point
and contain navigation, a command bar, single-line and multiline text input,
buttons, a toggle, list scrolling, theme selection, open/save, copy/paste, a
native menu, resize handling, scale-factor handling and keyboard focus
traversal.

## Stable Boundaries

The public desktop boundary consists of these safe Rust traits:

- `DesktopHost`
- `WindowService`
- `ClipboardService`
- `FileDialogService`
- `MenuService`
- `ThemeService`
- `TextInputService`

Every operation must have an explicit `DesktopCapability`. `Partial` and
`Unsupported` capabilities cannot be treated as runtime success. Raw `HWND`,
Objective-C objects, GTK pointers, drawing handles and platform event-loop
types remain private to backend modules.

Backend selection uses independent Cargo features:

- `windows-win32`
- `macos-appkit`
- `linux-gtk`
- `desktop-native` for the combined target-selected profile

Heavy backend dependencies must remain target-specific and optional.

## Backend Requirements

### Windows

Keep the existing Win32 host and buffered paint path. Complete and verify
window lifecycle, dynamic DPI, IME, native menus, clipboard, open/save dialogs,
system theme, focus traversal and resize repaint behavior. Background erase
suppression and buffered drawing are release invariants.

### macOS

Implement a real AppKit host around `NSApplication` and `NSWindow`, including
the native event loop, pointer/keyboard translation, Retina scaling, system
theme, `NSMenu`, `NSOpenPanel`, `NSSavePanel`, pasteboard access and basic
text/IME behavior. A Winit window is not AppKit completion evidence.

### Linux

Implement a real GTK4 host around `GtkApplication` and
`GtkApplicationWindow`, including event controllers, Wayland execution, basic
X11 compatibility, system theme, GIO/GTK menus, file chooser, clipboard and
basic text/IME behavior. GTK types do not enter the public widget API.

## Evidence Gate

A backend is a real runtime path only when all three evidence layers pass:

1. Compile: formatting, no-default tests, full tests, backend feature check and
   showcase example check on the target OS.
2. Runtime: startup, dark-theme, file-dialog, text-input and menu PNG artifacts
   under `docs/platform-proof/<platform>/`.
3. Interaction: a machine-readable report confirms lifecycle, typed button
   messages, text selection/copy/paste, basic CJK IME stability, scrolling,
   Tab focus, open/save, system-theme reaction and clean resize repaint.

`scripts/check-desktop-proof.ps1 -RequireComplete` is the final evidence gate.
It must not pass through placeholders or empty files.

## Current Baseline

| Area | Status | Meaning |
| --- | --- | --- |
| Safe desktop service traits | In progress | Public contracts and strict capability checks exist |
| Shared showcase | In progress | Shared state/view/update, native Win32 menu and smoke path exist |
| Windows Win32 | First pass | Buffered rendering, typed input, focus, scroll, owned menus, dynamic-DPI relayout, theme repaint, common dialogs and basic IMM32 commit exist; target proof remains incomplete |
| macOS AppKit | Not started | Current runtime remains the first-pass Winit window path |
| Linux GTK4 | Not started | Current runtime remains the first-pass Winit window path |
| Three-layer proof | Not complete | The manifest and validator exist; complete target artifacts do not |

Status changes require code, target execution and committed evidence. A feature
name, declaration, mock or cross-compile alone cannot advance a backend to
complete.

## Milestones

1. Stabilize desktop service contracts, shared events and capability tests.
2. Close the Win32 service loop without replacing the buffered renderer.
3. Connect the real AppKit host and record macOS evidence.
4. Connect the real GTK4 host and record Wayland/X11 evidence.
5. Enforce all three platform proof sets in CI and declare v0.2 complete.
